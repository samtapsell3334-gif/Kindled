# Kindled — Engineering Conventions

## Stack

| Layer | Technology |
|---|---|
| Framework | Next.js 15 (App Router, `"use client"` at leaf boundaries only) |
| Language | TypeScript 5 · `exactOptionalPropertyTypes: true` |
| Styling | Tailwind CSS v4 (`@import "tailwindcss"` / `@theme` block) |
| Animation | Framer Motion |
| Icons | Lucide React (SVG only — no emoji in code) |
| Database | PostgreSQL via **Prisma** (schema at `prisma/schema.prisma`) |
| Payments | **Stripe Connect (Express)** — see `src/lib/stripe.ts` |
| Email | Resend — server-side only via `src/app/api/signup/route.ts` |
| AI Video | Kling AI (default) / Runway ML — see `src/lib/ai-reveal.ts` |

## Naming conventions

- **Pot** = a "Fire" in product language. All code uses `Pot` / `pot` to match Prisma models.
- **Contribution** = a "Kindle" or chip-in payment. Code uses `Contribution`.
- **Host** = the pot creator (receiver of the gift). Code uses `creator` / `creatorId`.
- **Giver** = a contributor. Code uses `giver` / `giverId`.

## Directory structure

```
src/
  app/           # Next.js App Router pages and API routes
    api/         # Server-only route handlers
  components/    # Shared UI components ("use client" where needed)
  lib/           # Pure business logic and service clients (no React)
    db.ts        # Prisma client singleton
    stripe.ts    # Stripe Connect initialisation
    schema.ts    # TypeScript type layer (mirrors Prisma models)
    fees.ts      # Fee calculation — financial unit tests required here
    pot-transitions.ts  # Pure status-transition logic
    ai-reveal.ts # AI video generation service
  types/         # Shared TypeScript interfaces
prisma/
  schema.prisma  # Authoritative data model
  migrations/    # Auto-generated by `prisma migrate dev`
```

## Financial logic rules

1. **All monetary values use `Decimal` from `@prisma/client/runtime/library`** — never `number` or `float` for money.
2. **Rounding rule**: `ROUND_HALF_UP` on each fee component independently so that components always sum to gross.
3. **Fee structure** (see `src/lib/fees.ts`):
   - Platform fee: 0.5% of gross
   - Processing cost: £0.05 flat (A2A pass-through)
   - `netToPot = gross − platformFee − processingCost`
4. **Unit tests required** for any function that touches amounts, fees, or status transitions.
5. Never perform arithmetic on `Decimal` values using JS `+`, `-`, `*`. Use `.add()`, `.sub()`, `.mul()`, `.div()`.

## Status machine

`Pot.status`: `ACTIVE → HALFWAY → FUNDED` (irreversible). `STALLED` is set only by a background job, never by a webhook.

See `src/lib/pot-transitions.ts` — `resolveTransition()` is the single source of truth.

## Stripe conventions

- Always use **Stripe Connect Express** for payment flows.
- The platform's Stripe secret is `STRIPE_SECRET_KEY` (server-side only).
- The publishable key exposed to the client is `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`.
- Webhook secret is `STRIPE_WEBHOOK_SECRET`.
- Connected account IDs are stored on `User.stripeAccountId`.
- Use idempotency keys on every `paymentIntent.create` call: `potId:giverId:amountMinorUnits`.

## Stripe Connect flow

```
1. Host onboards → Stripe Express account created → stripeAccountId saved on User
2. Giver initiates contribution → PaymentIntent created on the connected account
3. Webhook (stripe-signature verified) → Contribution status updated → pot balance recalculated
4. Platform fee collected via application_fee_amount on the PaymentIntent
```

## Security rules

- `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `DATABASE_URL`, `RESEND_API_KEY`, `CREATOR_SIGNUP_EMAIL`, `KLING_API_KEY`, `RUNWAY_API_KEY` are **server-side only** — never in client bundles or `NEXT_PUBLIC_` vars.
- Verify `stripe-signature` on every incoming webhook before processing.
- Run `db.pot.findUnique` ownership checks before any mutation — never trust client-supplied `creatorId`.
- Privacy boundary on surprise pots: receivers see zero balance until `eventDate`.

## Code quality

- **No emoji in source code** — use Lucide icons or CSS primitives.
- **No `any` casts** unless bridging an ungenerated Prisma model (document why in a comment).
- **No unused imports** — the ESLint config treats them as errors.
- `exactOptionalPropertyTypes: true` — use spread (`{ ...(x ? { key: x } : {}) }`) instead of setting optional fields to `undefined`.
- Business logic lives in `src/lib/` and is independently testable.
- API routes live in `src/app/api/` and import from `src/lib/` — no logic inline.
- Components in `src/components/` may not import from `src/app/`.
